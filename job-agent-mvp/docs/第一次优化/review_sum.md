## 2. 第一次优化分析过程

### 2.1 优化前：系统已经“能跑”，但离真实需求还有明显距离

在第一次优化前，系统最大的问题并不是流程不通，而是“虽然能输出结果，但结果还不够可信”。
总览文档对这个状态总结得很准确：当前系统已是**形式稳定的 MVP**，但还不是**事实稳健的 MVP**。问题集中在四个模块：

* `candidate_profile`：更像“切文本”，不是真正结构化
* `jd_analysis`：更像“总结”，不是可驱动下游的拆解
* `match_analysis`：词匹配偏多，能力匹配偏少，容易过度推断
* `rewrite`：表达变强了，但事实边界没有守死

这说明系统已经具备“工作流外形”，但离用户真实需求还差一层关键能力：

> **中间表示要干净，判断要有证据，生成要守边界。** 

---

### 2.2 优化目标：从“流程跑通”转向“缩短需求差距”

第一次优化本质上不是单点修 bug，而是在围绕用户真实需求缩短差距：

| 用户需求       | 优化前问题            | 优化目标          |
| ---------- | ---------------- | ------------- |
| 正确理解候选人经历  | profile 更像切文本    | 建立可信中间表示      |
| 正确理解 JD 要求 | JD 更像总结          | 转成可驱动下游的结构化标准 |
| 正确判断匹配关系   | match 过度依赖词匹配    | 建立证据分级和验证边界   |
| 安全改写       | rewrite 会虚构数字/功能 | 建立事实白名单和自查机制  |

---

### 2.3 具体分析过程：每个模块如何缩短“结果”和“需求”的距离

#### 2.3.1 candidate_profile：从“切文本”到“中间表示”

优化前，`skills / projects / experiences` 三个字段高度重叠，`skills` 中混入大量完整句子，`projects` 和 `experiences` 几乎重复，`missing_items` 失效，`summary` 只是按行数统计，不是真正的语义单元统计。文档明确指出：当前 `candidate_profile` 更像“切文本”，不是“真正结构化”。

这与用户需求的差距在于：

* 用户需要的是“候选人事实层”
* 系统给出的却是“被切碎的原文片段”

因此第一次优化的核心动作是：

* 要求 `skills` 只保留独立技能点
* 将 `projects` 结构化为对象，包含 `name / period / role / tech_stack / highlights`
* 删除无价值重复字段，或将其明确区分为 `work_experiences`
* 增加 `missing_items` 检查清单
* 将 schema 从扁平结构升级为嵌套结构

优化后的结果是：

* `skills` 从混入整句的 38 项，变成了 18 项独立技能点
* `projects` 从 8 条逐行切割文本，变成 2 个结构化项目对象
* `missing_items` 变成有意义的缺失项
* 新增 `basic_info` 和 `work_experiences`
* `projects` 可以直接作为 rewrite 的事实白名单输入

这一步缩短的不是“格式差距”，而是：

> **把候选人材料从杂乱原文，变成了可供下游稳定使用的事实层。**

---

#### 2.3.2 JD 分析：从“总结 JD”到“拆解 JD”

优化前，JD 模块虽然能提取 responsibilities、required_skills、keywords 和 candidate_focus，但存在明显问题：

* `keywords` 混合了岗位名、工具名、能力词、概念词
* `required_skills` 是大段原文，不是独立可评估单元
* `candidate_focus` 没有优先级
* 本质上更像总结，不是可直接驱动下游的拆解

这与用户需求的差距在于：

* 用户需要知道“岗位真正看重什么”
* 下游模块需要逐条判断“候选人是否满足具体要求”
* 但原始输出不能直接支撑这类判断

因此第一次优化做了三类关键动作：

* 按 `tech / capability / tool / domain` 分桶输出 keywords
* 将 `required_skills` 结构化为 `skill + priority + verifiable + evidence_hint`
* 为 `candidate_focus` 增加优先级字段，并补充 `jd_risk_flags`

优化后的结果是：

* `required_skills` 从 9 条原文大段话，变成 20 条独立 SkillItem
* `keywords` 从无分类列表，变成 4 个维度的分桶结构
* `candidate_focus` 增加优先级和 reason
* 新增 `jd_risk_flags`
* JD 输出开始可以直接驱动 match 模块逐条评估

这一步缩短的核心差距是：

> **让系统从“读懂 JD”进化到“把 JD 变成可执行标准”。**

---

#### 2.3.3 match：从“词命中”到“证据判断”

优化前，match 模块最大的问题是“从弱证据推强结论”。
例如，把“分层存储架构”直接推成“任务拆解与分层设计能力”，把“检索隔离”推成“具备数据契约与边界治理意识”。文档明确指出：当前 match 分析是“词匹配偏多，能力匹配偏少”，且没有区分“简历里提到了什么”和“能力真的成立到什么程度”。

这与用户需求的差距在于：

* 用户要的是可信判断
* 系统给出的却是未经分级的乐观推断

因此第一次优化采取的核心动作是：

* 引入 `evidence_level: strong | medium | weak | inferred`
* 明确禁止从单一功能点推断综合能力
* 增加 `needs_verification`
* 将 `high_match_points` 升级为结构化 MatchPoint，包含 `point / evidence / evidence_level / jd_requirement`

优化后的结果是：

* `high_match_points` 从 6 条纯字符串变成 7 条结构化 MatchPoint
* 证据等级分布明确为 strong×4、medium×3
* `needs_verification` 新增 5 条
* `rewrite_focus` 不再要求候选人去“补充”自己并不具备的能力
* 模块结论从“容易过度推断”变成“基本满足”，虽然仍有个别映射略牵强，但已经有 `needs_verification` 兜底

这一步缩短的核心差距是：

> **把匹配分析从“关键词碰撞”推进成“证据驱动的能力判断”。**

---

#### 2.3.4 rewrite：从“更强表达”到“安全改写”

rewrite 是优化前风险最高的模块。
当时的改写策略虽然写着“保持真实性”，但实际输出里仍然出现了虚构量化数字、脑补功能和能力拔高，例如虚构“降低串库风险约30%”“提升回答相关性约20%”，甚至补入“支持后续监控告警与自动恢复机制集成”等原文不存在的内容。文档将其定义为典型的“策略声明”和“实际行为”不一致问题。

这与用户需求的差距在于：

* 用户需要的是“能放心复制进简历的内容”
* 系统却产出了“看起来更强、但用户未必敢认”的表述

因此第一次优化的关键动作是：

* 明确禁止生成任何原文未出现的数字、百分比、量化指标
* 禁止添加原文未提及的功能、技术或集成
* 禁止用 JD 中的概念替换候选人自己的描述
* 将 rewrite 的输入改造成“事实白名单 + JD”
* 引入 `added_facts` 自审机制，对新增内容进行显式声明

优化后的结果非常明确：

* 虚构量化数字从 4 处降到 0 处
* 脑补功能从 2 处降到 0 处
* 能力拔高消失，改为保留原文表述并做结构优化
* `added_facts` 新增，且本次输出为空列表
* 模块结论为“完全满足”

这一步缩短的核心差距是：

> **把简历改写从“会生成”变成“可安全使用”。**

---

### 2.4 第一次优化的整体结论

第一次优化的价值，不在于某个模块“更聪明了”，而在于它让整个系统开始接近真正的产品需求：

* `candidate_profile` 建立了候选人事实层
* `jd_analysis` 建立了岗位要求标准层
* `match` 建立了证据分级的判断层
* `rewrite` 建立了事实边界内的表达层

因此，这次优化最准确的总结不是“优化了 prompt 和 schema”，而是：

> **把项目从“流程跑通的 Agent MVP”推进到了“开始具备事实约束能力的 Workflow Agent MVP”。**

不过，这次优化还没有完全结束所有问题。当前已知遗留问题是：简历解析器在软换行处仍可能错误切分句子，说明 parser 的鲁棒性还需要继续增强。
