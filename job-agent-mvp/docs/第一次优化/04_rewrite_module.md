## 问题分析

### 哪些改写是合理的

- ✅ 将"分层存储架构"的描述更清晰地按职责分工呈现（PostgreSQL/MongoDB/Milvus 各自职责）
- ✅ 将 RAG 链路描述得更完整（文档处理→向量检索→流式问答）
- ✅ 增加了"AI原生架构"等 JD 关键词的自然融入
- ✅ 对缺乏数据的量化结果标注了"需补充具体数据"

### 哪些改写存在问题

**1. 生成了未经证实的量化数字**

- `"降低串库风险约30%"` — 原文没有任何数字，这是凭空生成的
- `"提升回答相关性约20%"` — 原文没有任何数字，这是凭空生成的
- `"提升投递效率约15%"` — 原文没有任何数字，这是凭空生成的
- `"减少手动修改时间约25%"` — 原文没有任何数字，这是凭空生成的

虽然加了"需补充具体数据"的标注，但这些数字本身不应该出现在改写结果里。候选人如果直接复制使用，就会在简历中出现无法支撑的数据。

**2. 引入了原文没有的技术描述**

- `"支持工具调用（如AI编程工具集成）"` — 原文 Agent MVP 项目没有提到工具调用或 AI 编程工具集成，这是脑补的
- `"支持后续监控告警与自动恢复机制集成"` — 原文没有提到监控告警，这是为了对齐 JD 关键词而添加的未实现功能

**3. 能力描述被拔高**

- `"构建企业级AI原生架构"` — 原文是"开发企业级多租户智能知识库与问答平台"，"AI原生架构"是 JD 的概念，不是候选人自己的描述
- `"形成智能体式处理闭环"` — 原文有类似描述，但"智能体式"这个措辞是 JD 风格，不是候选人原文

## 本质问题

> 输出表达变强了，但事实边界没有守死

系统做的事：在"保持真实性"的策略声明下，实际上做了量化数字生成和功能脑补。
系统应该做的事：只强化表达方式，不新增任何事实（包括数字、功能、技术描述）。

这是一个典型的 **"策略声明"和"实际行为"不一致** 的问题。

---

## 优化方向

### 短期（prompt 层面）

**最关键的一条禁止规则：**

```
禁止生成任何原文中未出现的数字、百分比、量化指标。
如果原文没有数据，不要生成占位数字，直接跳过量化表达。
```

**其他禁止规则：**

```
禁止添加原文未提及的功能、技术或集成（即使是"支持后续..."这类表述）。
禁止使用 JD 中的专有概念替换候选人自己的描述（如不要把"开发平台"改成"构建AI原生架构"）。
```

**允许的操作：**

```
允许：重新组织句子结构，使表达更清晰
允许：将技术栈名称标准化（如"Spring AI"保持不变）
允许：将多个相关描述合并为一条更完整的描述
允许：在原文有明确描述的基础上，使用更专业的技术术语
```

### 中期（输入改造）

将 rewrite 的输入从"原始简历文本 + JD"改为"事实白名单 + JD"：

```json
{
  "fact_whitelist": {
    "project_name": "IntelliVault",
    "tech_stack": ["Java 17", "Spring Boot 3", "Spring AI", "PostgreSQL", "MongoDB", "Milvus", "Docker"],
    "verified_facts": [
      "实现了文档处理→向量检索→SSE流式问答的完整RAG链路",
      "设计了PostgreSQL/MongoDB/Milvus分层存储架构",
      "实现了检索隔离与向量生命周期管理",
      "基于JWT实现无状态接口鉴权",
      "将会话消息异步写入存储系统"
    ],
    "no_data_available": true,
    "forbidden_additions": ["监控告警", "自动恢复", "工具调用", "AI编程工具集成"]
  }
}
```

### 长期（验证机制）

增加 rewrite 输出的事实核查步骤：

1. 对改写后的每条描述，检查是否能在原文中找到对应的事实依据
2. 对所有数字进行标记，如果原文没有对应数字，直接拒绝输出
3. 增加 `added_facts` 字段，让系统自我报告新增了哪些内容（便于人工审核）

### 核心认识

> rewrite 的目标是"让真实的能力被更好地看见"，不是"让简历看起来更强"。
> 一旦越界生成，候选人在面试中就会被追问无法回答的问题，反而造成负面影响。
> fallback 的目标不是"聪明"，而是"可信"。
