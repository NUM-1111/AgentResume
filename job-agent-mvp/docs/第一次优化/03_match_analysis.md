# match 分析模块复盘

> 本文档聚焦简历匹配度分析模块的实际输出质量、存在问题与优化方向。

---

## 本次实际输出（真实结果）

### 高匹配点（high_match_points）

```
1. 具备RAG系统实际落地经验：在IntelliVault项目中实现了完整的文档处理、向量检索、Prompt约束和流式问答链路，符合JD对RAG/知识增强模块的要求。
2. 有Agent项目开发经验：独立开发了基于大模型的求职场景智能助手，实现了任务拆解、Prompt模板设计、多轮交互优化等功能，展示了Agent系统的基本构建能力。

### 明确缺口（gap_items）

```
1. 缺乏AI原生架构设计经验：JD强调从零设计AI原生架构，候选人项目更多是现有框架（Spring AI）的应用，缺乏从零设计的完整案例。
2. 缺乏高可用架构与监控告警实践经验：候选人材料中未涉及系统监控、告警、自动恢复等相关内容。
3. 缺乏分层验证体系与CI阻断机制经验：候选人材料中未展示相关实践经验。

6. 缺乏社区影响力展示：候选人材料中未提供GitHub项目、技术博客等相关证据。

### 风险点（risk_points）

```
1. 技术栈偏向后端Java生态：候选人主要使用Java+Spring技术栈，而AI算法工程师岗位可能更偏向Python生态，存在技术栈匹配风险。
2. 项目时间较短：IntelliVault项目从2025.12开始至今（假设当前为2026年初），经验可能不足半年；求职助手项目仅2个月，深度可能有限。
3. 缺乏大规模线上系统经验：候选人项目可能为学习或小规模实践项目，缺乏高并发、高可用生产环境的实战经验。
4. 对大模型局限性的理解深度待验证：候选人材料中未详细阐述对模型局限性的认知和应对策略。
5. 独立复现先进项目经验不足：候选人材料中未展示相关能力，可能影响学习力评估。
```

## 问题分析

### 哪些判断是合理的

- ✅ RAG 落地经验的识别是准确的（IntelliVault 确实有完整 RAG 链路）
- ✅ Prompt Engineering 能力的识别是准确的（两个项目都有明确体现）
- ✅ 技术栈偏 Java 的风险识别是准确的
- ✅ 项目时间短的风险识别是准确的
- ✅ 缺乏监控告警、CI阻断、灰度发布的缺口识别是准确的

### 哪些判断存在问题

**1. 从弱证据推到强结论**

- "具备任务拆解与分层设计能力"：把"分层存储架构"（数据库设计）等同于"任务拆解能力"（Agent 规划能力），这是两个不同维度的能力，被混为一谈
- "具备数据契约与边界治理意识"：这个结论来自"检索隔离"这一个功能点，证据偏弱，但结论措辞较强

**2. 证据等级没有区分**

- 所有 high_match_points 都是同等权重呈现，但实际上：
  - "RAG 落地经验"有明确代码级证据（强证据）
  - "Agent 系统基本构建能力"只有功能描述（中等证据）
  - "数据契约与边界治理意识"只有一个功能点（弱证据）

**3. 没有区分"简历中有"和"能力真实存在"**

- 候选人写了"设计任务拆解流程"，但这是 Agent MVP 项目的一个功能描述，不等于候选人真正掌握了 Agent 规划推理的工程能力
- match 模块没有对这种"描述 vs 能力"的差距做出区分

---

## 本质问题

> 当前 match 分析是"词匹配偏多，能力匹配偏少"

系统做的事：找 JD 关键词在简历中是否出现，出现了就算匹配。
系统应该做的事：评估候选人是否真正具备 JD 要求的能力，区分证据强度。

---

## 优化方向

### 短期（prompt 层面）

1. 要求每条 match 结论标注证据等级：
   ```
   evidence_level: "strong" | "medium" | "weak" | "inferred"
   ```
   - `strong`：简历中有明确的代码/架构/功能实现描述
   - `medium`：简历中有相关功能描述，但细节不足
   - `weak`：只有一个功能点或间接相关
   - `inferred`：从其他能力推断，无直接证据

2. 禁止从单一功能点推断综合能力，在 prompt 中明确：
   > "不要把'实现了X功能'等同于'具备X领域的系统设计能力'"

3. 增加 `needs_verification` 字段，标注哪些匹配点需要面试验证

### 中期（schema 层面）

将 match 输出结构升级为：

```json
{
  "high_match_points": [
    {
      "point": "RAG系统实际落地经验",
      "evidence": "IntelliVault 项目中实现了文档处理、向量检索、流式问答完整链路",
      "evidence_level": "strong",
      "jd_requirement": "RAG/知识增强模块落地"
    }
  ],
  "weak_match_points": [...],
  "gap_items": [...],
  "risk_points": [...],
  "needs_verification": ["多agent协作能力", "大模型风险应对深度", "AI编程工具使用经验"]
}
```

### 核心认识

> match 模块的价值不是"给候选人打分"，而是"给 rewrite 模块提供可信的事实白名单"。
> 如果 match 过度推断，rewrite 就会在错误的基础上继续放大。
