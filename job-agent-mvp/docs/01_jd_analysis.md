## 问题分析

### 当前表现

- ✅ 能完整提取 responsibilities、required_skills、bonus_skills
- ✅ 能生成 keywords 列表
- ✅ 能生成 candidate_focus（投递建议方向）
- ⚠️ keywords 混杂了技术词、能力词、工具名、概念词，没有分层
- ⚠️ candidate_focus 是"建议"，但没有优先级排序
- ⚠️ required_skills 里每条都是一大段话，没有拆成独立的可评估单元

### 本质问题

> 当前 JD 分析更像"总结"，还不是"可直接驱动下游的拆解"

具体表现：
- `keywords` 里 `AI算法工程师`（岗位名）和 `vLLM`（具体工具）混在同一个列表，下游无法区分权重
- `required_skills` 每条是原文整句，没有拆成"技能点 + 重要程度 + 是否可验证"的结构
- 没有区分"硬性要求"和"加分项"的优先级差异
- `candidate_focus` 没有标注哪些是高优先级、哪些是锦上添花

---

## 优化方向

### 短期（prompt 层面）

1. 在 prompt 中要求 keywords 按类型分组输出：
   - `tech_keywords`：具体技术词（RAG、vLLM、LangChain）
   - `capability_keywords`：能力词（任务拆解、多agent协作）
   - `tool_keywords`：工具名（Cursor、Claude Code、Docker）

2. 要求 required_skills 每条输出结构化字段：
   ```json
   {
     "skill": "熟练使用AI编程工具",
     "priority": "high",
     "verifiable": true,
     "evidence_hint": "项目中是否提到使用 Cursor/Claude Code"
   }
   ```

3. candidate_focus 增加优先级字段（`priority: high/medium/low`）

### 中期（schema 层面）

- 在 JD schema 中增加 `skill_priority_map`，明确区分 must-have / nice-to-have
- 增加 `jd_risk_flags`：标注 JD 中对候选人可能构成风险的要求（如"从零设计AI原生架构"这类高门槛要求）

### 核心认识

> JD 分析的价值不在于"把 JD 读懂"，而在于"把 JD 转化成可以驱动 match 和 rewrite 的结构化输入"。
