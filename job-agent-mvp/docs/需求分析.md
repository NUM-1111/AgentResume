# Job Agent MVP 需求分析（具体化版本）

## 1. 总体需求定义（重构）

> **目标不是生成内容，而是构建一个“基于事实的求职决策与表达系统”**

系统核心职责：

* 将 **JD → 可执行标准**
* 将 **简历 → 事实结构**
* 将 **匹配 → 证据判断**
* 将 **改写 → 受约束表达**

---

## 2. 抽象需求 → 可执行需求映射

---

### 2.1 正确性需求（Correctness）

#### ❌ 抽象说法

> 正确理解 JD / 正确理解候选人

#### ✅ 具体化定义

##### （1）JD 正确性标准

系统必须做到：

* 将 JD 拆解为**独立 SkillItem**

  * 每一项包含：

    * `skill`
    * `priority（high / medium / low）`
    * `verifiable（是否可验证）`
    * `evidence_hint（如何验证）`

* keywords 必须**分类明确**

  * tech（技术）
  * capability（能力）
  * tool（工具）
  * domain（领域）

##### ✅ 判断是否达标

* 是否还能看到“整段 JD 原文”出现在 required_skills？
* 每个 skill 是否可以**单独被 match 判断？**

---

##### （2）候选人正确性标准

系统必须做到：

* skills：

  * 只能是**独立技能点**
  * ❌ 不能包含完整句子

* projects：

  * 必须结构化为对象：

    ```
    {
      name,
      period,
      role,
      tech_stack,
      highlights
    }
    ```

* experiences / projects：

  * ❌ 不能重复
  * ✅ 必须语义区分

##### ✅ 判断是否达标

* skills 是否可以直接用于匹配（而不是再解析）？
* projects 是否可以直接作为 rewrite 输入？

---

### 2.2 可信度需求（Trustworthiness）

#### ❌ 抽象说法

> 不要虚构、要可信

#### ✅ 具体化定义

##### （1）rewrite 约束规则（硬性）

系统必须保证：

* ❌ 禁止生成：

  * 数字（%、提升、降低等）
  * 原文未出现的功能
  * 原文未出现的技术

* ❌ 禁止行为：

  * 用 JD 词汇替换候选人事实
  * 能力拔高（例如：实现 → 设计 → 架构）

* ✅ 必须机制：

  * `added_facts` 字段

    * 所有新增内容必须记录
    * 理想状态：为空

##### ✅ 判断是否达标

* 改写内容是否“用户敢承认”？
* 是否存在任何“脑补信息”？

---

##### （2）match 可信度标准

系统必须做到：

* 每个匹配点必须包含：

  ```
  {
    point,
    evidence,
    evidence_level,
    jd_requirement
  }
  ```

* evidence_level：

  * strong（直接明确）
  * medium（间接支持）
  * weak（弱相关）
  * inferred（推断）

* ❌ 禁止：

  * 单一功能 → 推导综合能力

* ✅ 必须：

  * `needs_verification`

##### ✅ 判断是否达标

* 每个结论是否都有证据？
* 是否区分“提到过”和“真正具备”？

---

### 2.3 决策辅助需求（Decision Support）

#### ❌ 抽象说法

> 帮助用户判断是否匹配

#### ✅ 具体化定义

系统必须输出：

##### （1）match 结果结构

* high_match_points
* weak_match_points
* gap_items
* risk_points
* needs_verification

##### （2）rewrite 输入约束

* rewrite 只能使用：

  * high_match_points（strong / medium）
  * candidate_profile.projects

* ❌ 不允许使用：

  * weak / inferred 能力

##### ✅ 判断是否达标

* 用户是否可以回答：

  * “我为什么适合这个岗位？”
  * “我哪里不匹配？”
  * “面试官会问我哪里？”

---

### 2.4 效率需求（Efficiency）

#### ❌ 抽象说法

> 提高效率

#### ✅ 具体化定义

##### （1）系统层

* Pipeline 固定流程：

  ```
  JD → Profile → Match → Rewrite
  ```

* 每一步输出：

  * 必须结构化（JSON）
  * 必须可复用

##### （2）工程层

* schema 校验（Pydantic）
* fallback 机制（无 LLM 可运行）
* 日志记录（JSONL）

##### （3）中间表示复用

* JDAnalysis → 直接驱动 Match
* Profile → 直接驱动 Rewrite

##### ✅ 判断是否达标

* 是否有模块重复解析原始文本？
* 是否可以单独复用某一步输出？

---

## 3. 当前系统 vs 理想需求（差距拆解）

| 模块                | 当前问题  | 对应需求缺失 | 优化方向           |
| ----------------- | ----- | ------ | -------------- |
| candidate_profile | 切文本   | 无事实层   | 建立结构化 profile  |
| jd_analysis       | 总结 JD | 无执行标准  | SkillItem 化    |
| match             | 词匹配   | 无证据体系  | evidence_level |
| rewrite           | 会编造   | 无事实边界  | added_facts    |

---

## 4. 关键结论（可用于面试表达）

> 本项目的核心优化，不是提升生成能力，而是：

* 建立 **中间表示（结构化事实 & 标准）**
* 引入 **证据机制（防止过度推断）**
* 强化 **事实边界（防止幻觉）**

最终实现：

> **从“生成型 Agent” → “可控推理型 Workflow Agent”**

* JD → 行业知识增强（不是纯文本拆解）
