# Job Agent MVP 第一次优化复盘：从“流程跑通”到“输出可信”

## 1. 用户的真实需求

### 1.1 项目表面需求

从产品流程上看，这个项目是一个面向求职场景的 AI Pipeline Agent，输入 `JD + 候选人背景文本 + 目标岗位方向`，自动完成 **JD 解析 → 简历结构化 → 匹配分析 → 简历改写**，输出结构化结果。项目的核心约束是：**不捏造事实，所有改写必须有原始依据**。

### 1.2 用户真正需要的并不是“自动生成”

如果只从功能描述理解，这个项目像是在做“自动改简历”。
但从测试场景和模块设计来看，用户真正需要的并不是一个会生成内容的系统，而是一个能够：

- 正确理解 JD 重点
- 正确理解候选人已有经历
- 正确判断“经历”和“岗位要求”之间的匹配关系
- 在不越界、不虚构的前提下优化表达

也就是说，这个项目真正解决的问题不是“替用户写内容”，而是：

> **把求职者已有事实，转化成针对目标岗位的可信表达。**

这一点从项目现阶段结论里也很清楚：系统已经是**形式稳定的 MVP**，但还不是**事实稳健的 MVP**；当前核心问题集中在**分析质量**，而不是流程是否跑通。

### 1.3 真实用户需求可以拆成四类

#### （1）正确性需求

用户希望系统输出的内容不只是“看起来合理”，而是真能支撑求职判断。
例如，JD 分析不能只是总结原文，而必须转成可以驱动后续 match 和 rewrite 的结构化输入。JD 模块的核心价值，不在于“把 JD 读懂”，而在于“把 JD 转化成可以驱动 match 和 rewrite 的结构化输入”。

#### （2）可信度需求

这是本项目最核心的需求。
项目明确要求改写模块不能捏造数字、功能、技术，并通过 `added_facts` 自审字段约束新增事实。
这说明用户真正关心的不是“简历写得多强”，而是“写出来的内容自己敢不敢在面试里承认”。

#### （3）决策辅助需求

用户不是为了看一份漂亮分析报告，而是要用输出结果做后续动作，例如：

- 这个岗位值不值得投
- 哪些经历可以重点强调
- 哪些能力只是弱匹配，面试里容易被追问
- 哪些地方不能写，因为没有事实依据

所以 match 模块的价值也不是“给候选人打分”，而是“给 rewrite 模块提供可信的事实白名单”。

#### （4）效率需求

用户需要的是一个可以复用的工作流，而不是一次性的 prompt 实验。
项目通过 orchestrator 串联 JD 分析、候选人结构化、match、rewrite 四个步骤，并配合 schema 校验、日志和 fallback，说明它要满足的是一个可复用、可稳定执行的求职工作流需求。

### 1.4 最终需求定义

因此，这个项目最准确的需求定义可以写成：

> **这是一个面向求职场景的 Workflow Agent，它不是为了自动生成“更好看”的简历，而是为了在真实经历不变的前提下，完成 JD 拆解、候选人事实结构化、证据驱动的匹配分析，以及安全改写。**
